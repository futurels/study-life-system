## 2. 数据库设计背景

随着学习任务逐渐增多，个人在日常学习和生活管理中，常常会遇到计划不清晰、执行不连续、生活记录零散、复盘不及时等问题。为了解决这些问题，本系统以“计划—执行—记录—复盘—统计”为业务主线，构建一个面向个人用户的学习与生活综合管理平台。

数据库作为系统的核心基础层，需要能够稳定地支撑以下业务需求：

1. 用户注册、登录与个人资料维护；
2. 学习计划的新增、编辑、删除、查询和状态跟踪；
3. 日常生活记录的持续保存与按日查询；
4. 每日复盘内容的记录与总结；
5. 周统计、月统计等数据分析查询。

因此，本数据库设计的核心目标，不仅是满足当前业务功能，还要保证结构清晰、便于扩展、方便后续接口开发与系统维护。

---

## 3. 数据库设计目标

本数据库设计主要围绕以下目标展开：

### 3.1 满足核心业务需求

数据库应完整支撑系统第一版中的四大核心模块：

- 用户模块
- 学习计划模块
- 生活记录模块
- 每日复盘模块

### 3.2 保证数据结构清晰

表结构命名、字段命名、约束命名、索引设计应规范统一，保证后续开发人员可以快速理解系统数据结构。

### 3.3 保证数据安全与完整性

通过主键、唯一约束、外键约束、检查约束、默认值、逻辑删除标记等机制，尽可能减少脏数据、重复数据与非法数据的出现。

### 3.4 保证良好的扩展性

系统后续可能扩展图片上传、番茄钟、提醒通知、分类管理、AI 复盘等功能，因此数据库设计不能过于死板，应预留适度扩展空间。

## 4. 技术选型与设计原则

## 4.1 数据库选型

本系统数据库建议采用 **MySQL 8.0+**，原因如下：

1. MySQL 生态成熟，适合中小型管理系统开发；
2. 与 Spring Boot + MyBatis 技术栈兼容性良好；
3. 支持事务、索引、外键、检查约束等数据库能力；
4. 便于后续部署、测试和论文展示。

### 4.2 字符集设计

- **字符集：** `utf8mb4`
- **排序规则：** `utf8mb4_0900_ai_ci`

选择 `utf8mb4` 的主要原因是能够完整支持中文、英文、表情符号等多种字符，避免出现乱码或字符截断问题。

### 4.3 设计原则

数据库设计遵循以下原则：

#### （1）业务导向原则

表结构围绕系统真实业务需求设计，避免为了“看起来复杂”而引入不必要的表或字段。

#### （2）结构清晰原则

表名、字段名采用统一规范，尽量做到见名知意，减少理解成本。

#### （3）适度冗余原则

在满足规范化设计的基础上，为提高查询效率和开发便利性，可保留少量合理冗余字段，但不重复存储统计结果。

#### （4）可扩展原则

第一版只保留最核心的数据结构，但在用户资料、安全鉴权、扩展能力上预留接口空间。

#### （5）安全性原则

用户密码必须加密保存，业务数据必须按用户隔离，所有核心业务表建议采用逻辑删除机制，防止误删后无法恢复。

## 5. 业务模块与数据实体划分

结合系统需求分析，数据库第一版主要涉及以下核心数据实体：

| 模块名称 | 数据实体 | 说明 |
|---|---|---|
| 用户模块 | `sys_user` | 存储用户账号、密码、个人资料、状态信息 |
| 学习计划模块 | `study_plan` | 存储用户每日学习任务及执行状态 |
| 生活记录模块 | `life_record` | 存储用户每天的生活日志、心情、睡眠、饮食、运动信息 |
| 每日复盘模块 | `daily_review` | 存储用户对某一天的学习与生活总结 |
| 认证扩展模块 | `auth_refresh_token` | 存储刷新令牌及登录会话扩展信息 |

其中，统计模块第一版不单独设计统计表，而是通过对 `study_plan`、`life_record`、`daily_review` 三张业务表进行聚合查询实现，这样可以避免统计数据冗余和同步问题。

## 6. 概念结构设计（ER 思想说明）

从业务关系上看，本系统主要围绕“用户”这一核心实体展开，其他业务数据均依附于用户存在。

### 6.1 实体关系说明

1. 一个用户可以拥有多条学习计划；
2. 一个用户可以拥有多条生活记录；
3. 一个用户可以拥有多条每日复盘；
4. 一个用户可以拥有多个刷新令牌；
5. 一条学习计划只能属于一个用户；
6. 一条生活记录只能属于一个用户；
7. 一条每日复盘只能属于一个用户；
8. 一条刷新令牌只能属于一个用户。

### 6.2 主要关系类型

- `sys_user` 与 `study_plan`：一对多
- `sys_user` 与 `life_record`：一对多
- `sys_user` 与 `daily_review`：一对多
- `sys_user` 与 `auth_refresh_token`：一对多

### 6.3 日期维度约束设计

为了符合本系统“按天记录”的业务特点，设计中采用以下规则：

- `life_record`：同一个用户在同一天只能有一条生活记录；
- `daily_review`：同一个用户在同一天只能有一条每日复盘。

该设计便于前端按日期展示，也便于后端进行按月统计、按周统计和趋势分析。

## 8. 数据表详细设计

## 8.1 用户表 `sys_user`

### 8.1.1 表用途说明

用户表用于保存系统用户的基础信息、登录认证信息、账号状态等数据，是整个系统的基础表。

### 8.1.2 字段设计说明

| 字段名 | 类型 | 是否为空 | 默认值 | 说明 |
|---|---|---:|---|---|
| `id` | BIGINT UNSIGNED | 否 | 自增 | 用户主键ID，系统内部唯一标识 |
| `username` | VARCHAR(50) | 否 | 无 | 登录用户名，全系统唯一 |
| `password_hash` | VARCHAR(255) | 否 | 无 | 加密后的密码，不允许保存明文 |
| `nickname` | VARCHAR(50) | 否 | 无 | 用户昵称，用于界面展示 |
| `real_name` | VARCHAR(50) | 是 | NULL | 真实姓名，可选 |
| `gender` | TINYINT | 否 | 0 | 性别：0未知，1男，2女 |
| `email` | VARCHAR(100) | 是 | NULL | 邮箱地址，可用于找回密码 |
| `phone` | VARCHAR(20) | 是 | NULL | 手机号，可用于绑定账号 |
| `avatar_url` | VARCHAR(255) | 是 | NULL | 头像地址 |
| `bio` | VARCHAR(255) | 是 | NULL | 个人简介或签名 |
| `account_status` | TINYINT | 否 | 1 | 账号状态：0禁用，1正常 |
| `last_login_time` | DATETIME | 是 | NULL | 最近一次登录时间 |
| `created_at` | DATETIME | 否 | CURRENT_TIMESTAMP | 注册时间 |
| `updated_at` | DATETIME | 否 | CURRENT_TIMESTAMP | 最后更新时间 |
| `deleted_flag` | TINYINT | 否 | 0 | 逻辑删除标记：0未删除，1已删除 |

### 8.1.3 约束设计

- 主键：`id`
- 唯一约束：
  - `username` 唯一
  - `email` 唯一
  - `phone` 唯一
- 检查约束：
  - `gender` 只能为 0、1、2
  - `account_status` 只能为 0、1
  - `deleted_flag` 只能为 0、1

### 8.1.4 设计说明

用户表第一版采用单角色设计，不单独拆角色表和权限表。这样可以降低系统复杂度，更符合个人管理系统的业务定位。后续如果扩展管理员、审计员等角色，再补充角色表即可。

## 8.2 学习计划表 `study_plan`

### 8.2.1 表用途说明

学习计划表用于保存用户每天创建的学习任务，包括计划标题、内容、时间安排、优先级、完成状态等信息。

### 8.2.2 字段设计说明

| 字段名 | 类型 | 是否为空 | 默认值 | 说明 |
|---|---|---:|---|---|
| `id` | BIGINT UNSIGNED | 否 | 自增 | 学习计划主键ID |
| `user_id` | BIGINT UNSIGNED | 否 | 无 | 所属用户ID，关联用户表 |
| `plan_title` | VARCHAR(100) | 否 | 无 | 计划标题，如“学习 Java 集合” |
| `plan_content` | TEXT | 是 | NULL | 计划详细描述 |
| `plan_date` | DATE | 否 | 无 | 计划所属日期 |
| `start_time` | TIME | 是 | NULL | 计划开始时间 |
| `end_time` | TIME | 是 | NULL | 计划结束时间 |
| `priority_level` | TINYINT | 否 | 2 | 优先级：1高，2中，3低 |
| `plan_status` | TINYINT | 否 | 0 | 状态：0未开始，1进行中，2已完成，3已取消 |
| `completion_time` | DATETIME | 是 | NULL | 任务实际完成时间 |
| `remark` | VARCHAR(500) | 是 | NULL | 备注说明 |
| `created_at` | DATETIME | 否 | CURRENT_TIMESTAMP | 创建时间 |
| `updated_at` | DATETIME | 否 | CURRENT_TIMESTAMP | 更新时间 |
| `deleted_flag` | TINYINT | 否 | 0 | 逻辑删除标记 |

### 8.2.3 约束设计

- 主键：`id`
- 外键：`user_id` 关联 `sys_user.id`
- 检查约束：
  - `priority_level` 只能为 1、2、3
  - `plan_status` 只能为 0、1、2、3
  - `deleted_flag` 只能为 0、1
  - 当 `start_time` 和 `end_time` 同时存在时，`end_time >= start_time`

### 8.2.4 设计说明

学习计划表允许同一用户在同一天创建多条计划，这符合真实使用场景。例如用户可以在同一天同时安排“背英语单词”“刷 SQL 题”“看算法视频”等多个任务。

## 8.3 生活记录表 `life_record`

### 8.3.1 表用途说明

生活记录表用于保存用户某一天的日常日志信息，包括日记正文、心情、睡眠、饮食、运动等内容。

### 8.3.2 字段设计说明

| 字段名 | 类型 | 是否为空 | 默认值 | 说明 |
|---|---|---:|---|---|
| `id` | BIGINT UNSIGNED | 否 | 自增 | 生活记录主键ID |
| `user_id` | BIGINT UNSIGNED | 否 | 无 | 所属用户ID |
| `record_date` | DATE | 否 | 无 | 记录日期 |
| `diary_content` | TEXT | 否 | 无 | 日记正文内容 |
| `mood_status` | VARCHAR(20) | 是 | NULL | 心情状态 |
| `sleep_hours` | DECIMAL(4,1) | 是 | NULL | 睡眠时长，单位小时 |
| `diet_note` | VARCHAR(500) | 是 | NULL | 饮食简述 |
| `exercise_note` | VARCHAR(500) | 是 | NULL | 运动简述 |
| `created_at` | DATETIME | 否 | CURRENT_TIMESTAMP | 创建时间 |
| `updated_at` | DATETIME | 否 | CURRENT_TIMESTAMP | 更新时间 |
| `deleted_flag` | TINYINT | 否 | 0 | 逻辑删除标记 |

### 8.3.3 约束设计

- 主键：`id`
- 外键：`user_id` 关联 `sys_user.id`
- 唯一约束：`(user_id, record_date)`，保证同一用户同一天只有一条生活记录
- 检查约束：
  - `deleted_flag` 只能为 0、1
  - `sleep_hours` 为空时允许；不为空时必须在 0.0 到 24.0 之间

### 8.3.4 设计说明

生活记录按“一人一天一条”设计，便于：

1. 前端按日期查看当日日志；
2. 统计本月记录天数；
3. 避免用户同一天重复创建多条生活主记录造成数据混乱。

## 8.5 刷新令牌表 `auth_refresh_token`

### 8.5.1 表用途说明

刷新令牌表属于认证扩展表，用于支撑 JWT 登录后的刷新令牌管理、退出登录、令牌撤销、多端会话管理等安全能力。

### 8.5.2 字段设计说明

| 字段名 | 类型 | 是否为空 | 默认值 | 说明 |
|---|---|---:|---|---|
| `id` | BIGINT UNSIGNED | 否 | 自增 | 刷新令牌主键ID |
| `user_id` | BIGINT UNSIGNED | 否 | 无 | 所属用户ID |
| `token_uuid` | CHAR(36) | 否 | 无 | 令牌唯一编号，建议使用 UUID |
| `token_hash` | VARCHAR(255) | 否 | 无 | 刷新令牌的哈希值 |
| `expire_time` | DATETIME | 否 | 无 | 令牌过期时间 |
| `revoked_flag` | TINYINT | 否 | 0 | 是否已撤销：0否，1是 |
| `client_type` | VARCHAR(30) | 是 | NULL | 客户端类型，如 web、android |
| `client_ip` | VARCHAR(64) | 是 | NULL | 登录客户端 IP |
| `created_at` | DATETIME | 否 | CURRENT_TIMESTAMP | 创建时间 |
| `updated_at` | DATETIME | 否 | CURRENT_TIMESTAMP | 更新时间 |

### 8.5.3 约束设计

- 主键：`id`
- 外键：`user_id` 关联 `sys_user.id`
- 唯一约束：
  - `token_uuid` 唯一
  - `token_hash` 唯一
- 检查约束：
  - `revoked_flag` 只能为 0、1

### 8.5.4 设计说明

如果第一版系统只实现基础 JWT 登录，而不引入刷新机制，这张表可以先创建但暂不启用。保留该表的好处是，后续若需要实现“退出登录后令牌失效”“单设备登录”“刷新 access token”等功能时，不需要重新调整数据库结构。

## 9. 索引设计说明

为了保证系统的查询性能，数据库中对高频查询字段设计了合理索引。

### 9.1 用户表索引

- `username` 唯一索引：用于登录校验
- `email` 唯一索引：用于邮箱唯一性校验
- `phone` 唯一索引：用于手机号唯一性校验
- `account_status` 普通索引：用于快速筛选正常/禁用账号

### 9.2 学习计划表索引

- `(user_id, plan_date)`：用于按用户、日期查询任务列表
- `(user_id, plan_status, plan_date)`：用于按状态筛选学习计划
- `(user_id, priority_level, plan_date)`：用于按优先级筛选学习计划

### 9.3 生活记录表索引

- `(user_id, record_date)`：用于按用户和日期查询生活记录，同时支撑唯一约束

### 9.4 每日复盘表索引

- `(user_id, review_date)`：用于按用户和日期查询复盘记录，同时支撑唯一约束

### 9.5 刷新令牌表索引

- `user_id`：用于查询某个用户的令牌信息
- `expire_time`：用于定期清理过期令牌

## 10. 约束设计说明

数据库约束是保证数据正确性的关键机制，本系统主要采用以下约束形式：

### 10.1 主键约束

每张表均设置自增主键 `id`，用于唯一标识一条业务记录。

### 10.2 唯一约束

用于防止业务数据重复，主要包括：

- 用户表中的用户名、邮箱、手机号唯一；
- 生活记录表中 `(user_id, record_date)` 唯一；
- 每日复盘表中 `(user_id, review_date)` 唯一；
- 刷新令牌表中的 `token_uuid`、`token_hash` 唯一。

### 10.3 外键约束

所有核心业务表都通过 `user_id` 关联用户表，保证业务数据必须依附于真实存在的用户。

### 10.4 检查约束

通过 `CHECK` 约束对枚举值和数值区间进行限定，例如：

- 性别只能是 0、1、2；
- 学习计划状态只能是 0、1、2、3；
- 睡眠时长必须在 0 到 24 小时之间；
- 复盘评分必须在 1 到 10 分之间。

### 10.5 默认值约束

如 `created_at`、`updated_at`、`deleted_flag`、`account_status` 等字段设置默认值，减少插入数据时的空值问题。

## 11. 逻辑删除设计

本系统所有核心业务表均采用 `deleted_flag` 逻辑删除字段，设计原因如下：

1. 防止误删后数据不可恢复；
2. 便于后续做数据回收站、恢复功能；
3. 便于审计和历史查询。

### 11.1 删除规则建议

- `0`：未删除
- `1`：已删除

### 11.2 开发建议

后端在查询列表、详情时应统一增加条件：

```sql
deleted_flag = 0
```

这样可以保证逻辑删除的数据不参与正常业务展示。

## 12. 数据安全设计说明

为了满足系统安全需求，数据库设计中应重点关注以下内容：

### 12.1 密码安全

`password_hash` 字段只能保存加密后的密码摘要，严禁保存明文密码。推荐使用：

- BCrypt
- Argon2

### 12.2 数据隔离

所有学习计划、生活记录、每日复盘等业务数据，必须通过 `user_id` 与用户绑定。后端查询时必须带上当前登录用户的 `user_id` 条件，防止越权访问他人数据。

### 12.3 令牌安全

如果启用刷新令牌机制，数据库中建议保存刷新令牌的哈希值，而不是保存原始 token 内容，以降低泄露风险。

## 13. 统计模块的数据实现说明

统计模块是本系统的重要展示功能，但第一版不建议单独存储统计结果表，而应采用实时聚合查询方式实现。

### 13.1 原因说明

1. 统计数据本质来源于业务表，可通过 SQL 实时计算；
2. 若单独存统计表，后续业务变动时容易导致数据不一致；
3. 第一版系统数据量较小，实时统计不会造成明显性能压力。

### 13.2 可实现的典型统计项

1. 本周学习任务总数
2. 本周已完成任务数
3. 本周任务完成率
4. 本月生活记录天数
5. 本月复盘天数
6. 最近 7 天任务完成趋势

### 13.3 统计实现思路

- 学习任务统计：基于 `study_plan`
- 生活记录统计：基于 `life_record`
- 复盘统计：基于 `daily_review`

## 14. 数据库规范建议

为了保证后续代码与数据库的一致性，建议统一遵循以下规范：

### 14.1 命名规范

- 表名采用小写下划线命名，如 `study_plan`
- 字段名采用小写下划线命名，如 `plan_title`
- 约束名建议带前缀：
  - 主键：默认
  - 唯一约束：`uq_`
  - 外键约束：`fk_`
  - 检查约束：`ck_`
  - 索引：`idx_`

### 14.2 通用字段规范

核心业务表统一保留以下字段：

- `id`
- `created_at`
- `updated_at`
- `deleted_flag`

### 14.3 时间字段规范

- 日期：使用 `DATE`
- 日期时间：使用 `DATETIME`
- 单日时间段：使用 `TIME`
