## 4. 接口总体设计原则

### 4.1 RESTful 风格

系统接口统一采用 RESTful 风格设计，遵循以下原则：

- `POST`：用于新增数据或执行登录、退出等动作
- `GET`：用于查询数据
- `PUT`：用于整条更新或状态更新
- `DELETE`：用于删除数据（业务上采用逻辑删除）

### 4.2 接口统一前缀

系统所有后端接口统一使用以下前缀：

```text
/api
```

模块路径命名建议如下：

- 认证模块：`/api/auth`
- 用户模块：`/api/user`
- 学习计划模块：`/api/study-plan`
- 生活记录模块：`/api/life-record`
- 每日复盘模块：`/api/daily-review`
- 数据统计模块：`/api/statistics`

### 4.3 数据格式统一

除文件上传等特殊场景外，所有接口默认使用：

- **请求格式：** `application/json`
- **返回格式：** `application/json;charset=UTF-8`

### 4.4 字符编码统一

系统接口统一采用：

```text
UTF-8
```

以确保中文、英文及特殊字符在前后端传输过程中不出现乱码。

### 4.5 单用户数据隔离原则

系统第一版定位为个人成长管理系统，所有业务数据都必须与当前登录用户绑定。后端在处理学习计划、生活记录、每日复盘、统计查询时，必须基于当前登录用户 `userId` 查询，禁止通过前端传入任意用户 ID 来访问他人数据。

### 4.6 逻辑删除原则

学习计划、生活记录、每日复盘等核心业务表采用逻辑删除，接口删除操作仅更新 `deleted_flag = 1`，不执行物理删除。默认查询只返回 `deleted_flag = 0` 的数据。

## 5. 通用请求约束

## 5.1 通用请求头

### 5.1.1 未登录接口请求头

适用于注册、登录等无需鉴权的接口。

```http
Content-Type: application/json
Accept: application/json
```

### 5.1.2 已登录接口请求头

适用于查询个人信息、学习计划管理、生活记录管理、复盘管理、统计查询等接口。

```http
Content-Type: application/json
Accept: application/json
Authorization: Bearer {accessToken}
```

说明：

- `Authorization` 使用 JWT Bearer 模式；
- `{accessToken}` 为登录成功后服务端返回的访问令牌；
- 服务端应校验令牌有效性、过期时间和用户身份。

### 5.1.3 可选扩展请求头

可根据需要加入以下头信息：

```http
X-Request-Id: 请求唯一编号
X-Client-Type: web
```

用于链路追踪、日志排查或多端扩展。

---

## 5.2 参数传递约束

### 5.2.1 路径参数

适用于通过主键 ID 查询、修改、删除单条记录。

示例：

```text
GET /api/study-plan/{id}
PUT /api/life-record/{id}
DELETE /api/daily-review/{id}
```

要求：

- `id` 必须为正整数；
- 不允许传 0 或负数；
- 若记录不存在或不属于当前用户，应返回统一错误。

### 5.2.2 查询参数

适用于列表查询、条件筛选、分页查询、按日期范围查询等场景。

示例：

```text
GET /api/study-plan/list?pageNum=1&pageSize=10&planDate=2026-04-03&planStatus=2
```

### 5.2.3 请求体参数

适用于新增、编辑、登录、修改密码等操作，统一使用 JSON 结构。

---

## 5.3 时间与日期格式约束

系统接口涉及日期、时间、日期时间三类字段，应统一格式。

### 5.3.1 日期字段

用于计划日期、记录日期、复盘日期等字段。

```text
yyyy-MM-dd
```

示例：

```text
2026-04-03
```

### 5.3.2 时间字段

用于计划开始时间、结束时间。

```text
HH:mm:ss
```

示例：

```text
09:30:00
```

### 5.3.3 日期时间字段

用于创建时间、更新时间、完成时间、登录时间等。

```text
yyyy-MM-dd HH:mm:ss
```

示例：

```text
2026-04-03 21:35:00
```

说明：

- 前后端必须统一时间格式；
- 若后端使用 ISO-8601，也必须全局统一，不能部分接口使用 `T` 分隔、部分接口不使用。

---

## 5.4 分页参数约束

列表接口统一支持分页，建议参数如下：

| 参数名 | 类型 | 是否必填 | 默认值 | 说明 |
|---|---|---|---|---|
| `pageNum` | Integer | 否 | 1 | 当前页码，从 1 开始 |
| `pageSize` | Integer | 否 | 10 | 每页条数 |
| `sortBy` | String | 否 | createdAt | 排序字段 |
| `sortOrder` | String | 否 | desc | 排序方式：asc / desc |

分页约束建议：

- `pageNum >= 1`
- `pageSize` 最大不超过 `100`
- 未传分页参数时，后端应给默认值，避免一次性返回大量数据

分页返回结构统一如下：

```json
{
  "pageNum": 1,
  "pageSize": 10,
  "total": 25,
  "pages": 3,
  "list": []
}
```

## 6. 统一返回体约束

## 6.1 返回体统一结构

系统所有接口应统一返回以下结构：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

字段说明如下：

| 字段名 | 类型 | 说明 |
|---|---|---|
| `code` | Integer | 业务状态码，不建议直接等同于 HTTP 状态码 |
| `message` | String | 返回消息说明 |
| `data` | Object / Array / Null | 业务数据 |

### 6.1.1 成功返回示例

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {
    "id": 1
  }
}
```

### 6.1.2 分页成功返回示例

```json
{
  "code": 200,
  "message": "查询成功",
  "data": {
    "pageNum": 1,
    "pageSize": 10,
    "total": 25,
    "pages": 3,
    "list": []
  }
}
```

### 6.1.3 失败返回示例

```json
{
  "code": 400,
  "message": "请求参数错误",
  "data": null
}
```

## 6.2 HTTP 状态码建议

接口响应仍建议遵循常见 HTTP 状态码规范：

| HTTP 状态码 | 含义 | 说明 |
|---|---|---|
| 200 | OK | 请求成功 |
| 201 | Created | 新增成功（可选） |
| 400 | Bad Request | 请求参数错误 |
| 401 | Unauthorized | 未登录或令牌无效 |
| 403 | Forbidden | 无权限访问 |
| 404 | Not Found | 资源不存在 |
| 409 | Conflict | 数据冲突，如重复记录 |
| 500 | Internal Server Error | 服务器内部异常 |

说明：

- 实际项目中可统一返回 200，再通过 `code` 区分业务状态；
- 也可同时使用 HTTP 状态码和业务状态码，两者保持一致性更佳。

## 7. 统一错误码约束

建议定义统一错误码枚举，示例如下：

| 业务码 | 含义 | 使用场景 |
|---|---|---|
| 200 | 成功 | 通用成功 |
| 400 | 参数校验失败 | 请求参数格式错误、缺失、越界 |
| 401 | 未登录或令牌失效 | JWT 缺失、过期、非法 |
| 403 | 无权访问 | 访问他人数据、权限不足 |
| 404 | 数据不存在 | 根据 ID 查不到数据 |
| 409 | 数据冲突 | 用户名已存在、同一天记录已存在 |
| 422 | 业务规则校验失败 | 结束时间早于开始时间、评分越界等 |
| 500 | 系统内部异常 | 空指针、数据库异常、未知异常 |

建议扩展更细的业务码，例如：

| 业务码 | 说明 |
|---|---|
| 10001 | 用户名已存在 |
| 10002 | 用户名或密码错误 |
| 10003 | 原密码错误 |
| 10004 | 新密码与旧密码不能相同 |
| 11001 | 学习计划不存在 |
| 11002 | 学习计划不属于当前用户 |
| 11003 | 学习计划时间范围非法 |
| 12001 | 当日生活记录已存在 |
| 12002 | 生活记录不存在 |
| 13001 | 当日复盘已存在 |
| 13002 | 复盘记录不存在 |
| 14001 | 统计范围参数非法 |

## 8. 鉴权与安全约束

## 8.1 鉴权方式

系统第一版建议采用：

```text
JWT + Bearer Token
```

鉴权流程如下：

1. 用户通过登录接口提交用户名和密码；
2. 服务端校验成功后返回访问令牌；
3. 前端后续请求在请求头中携带 `Authorization: Bearer {token}`；
4. 后端统一拦截并解析用户身份；
5. 接口层禁止依赖前端传入的 `userId` 进行数据权限判断。

## 8.2 无需鉴权接口

以下接口默认无需登录：

- `POST /api/auth/register`
- `POST /api/auth/login`

如后续增加验证码、刷新令牌等接口，再按实际情况补充。

## 8.3 必须鉴权接口

以下接口必须登录后才能访问：

- 用户资料接口
- 学习计划接口
- 生活记录接口
- 每日复盘接口
- 数据统计接口
- 退出登录接口
- 修改密码接口

## 8.4 数据权限约束

后端必须遵循以下规则：

1. 用户只能查看自己的个人资料；
2. 用户只能新增、查询、修改、删除自己的学习计划；
3. 用户只能新增、查询、修改、删除自己的生活记录；
4. 用户只能新增、查询、修改、删除自己的每日复盘；
5. 所有统计结果必须仅统计当前登录用户的数据。

## 8.5 输入校验与安全要求

- 用户密码必须加密存储，严禁明文保存；
- 输入内容需做基础非空校验、长度校验、数值范围校验；
- 对文本内容需防止明显恶意输入；
- 删除接口必须校验资源归属；
- 更新接口必须校验记录存在且归属当前用户。

---

## 9. DTO 设计总约束

为避免接口请求体和数据库实体直接耦合，建议分层使用：

- `Request DTO`：用于接收前端请求
- `Response VO`：用于返回给前端展示
- `Entity`：用于数据库映射
- `Query DTO`：用于列表条件查询

约束如下：

1. 前端请求对象中不暴露 `deletedFlag`、`userId` 等服务端控制字段；
2. 服务端返回对象中不返回 `passwordHash`、`tokenHash` 等敏感字段；
3. 认证相关接口应单独定义返回对象；
4. 列表查询尽量使用 Query DTO 封装参数。

## 10. 各模块接口约束明细

## 10.1 认证模块接口

### 10.1.1 用户注册

- **接口路径：** `POST /api/auth/register`
- **接口说明：** 用户使用用户名和密码注册新账号
- **是否鉴权：** 否

#### 请求体

```json
{
  "username": "sakura",
  "password": "12345678",
  "confirmPassword": "12345678",
  "nickname": "樱",
  "email": "test@example.com"
}
```

#### 请求字段约束

| 字段名 | 类型 | 是否必填 | 约束说明 |
|---|---|---|---|
| `username` | String | 是 | 4~20 位，字母、数字、下划线，系统内唯一 |
| `password` | String | 是 | 8~20 位，建议包含字母和数字 |
| `confirmPassword` | String | 是 | 必须与 `password` 一致 |
| `nickname` | String | 是 | 1~50 位 |
| `email` | String | 否 | 格式合法，若填写则应唯一 |

#### 返回 data 建议

```json
{
  "userId": 1,
  "username": "sakura"
}
```

#### 业务约束

- 用户名不可重复；
- 邮箱如存在则不可重复；
- 密码与确认密码必须一致；
- 不返回密码相关信息。

---

### 10.1.2 用户登录

- **接口路径：** `POST /api/auth/login`
- **接口说明：** 用户使用用户名和密码登录系统
- **是否鉴权：** 否

#### 请求体

```json
{
  "username": "sakura",
  "password": "12345678"
}
```

#### 返回 data 建议

```json
{
  "accessToken": "jwt-token",
  "tokenType": "Bearer",
  "expiresIn": 7200,
  "refreshToken": "optional-refresh-token",
  "userInfo": {
    "id": 1,
    "username": "sakura",
    "nickname": "樱",
    "avatarUrl": null
  }
}
```

#### 业务约束

- 用户名不存在或密码错误时返回统一错误；
- 账号被禁用时禁止登录；
- 登录成功后可更新 `last_login_time`；
- 若第一版不实现刷新令牌，可不返回 `refreshToken`。

---

### 10.1.3 用户退出登录

- **接口路径：** `POST /api/auth/logout`
- **接口说明：** 当前登录用户退出系统
- **是否鉴权：** 是

#### 请求体

```json
{}
```

#### 业务约束

- 若启用刷新令牌机制，应撤销当前刷新令牌；
- 若未启用刷新令牌机制，该接口可仅做前端本地令牌清理提示。

---

### 10.1.4 刷新访问令牌（预留）

- **接口路径：** `POST /api/auth/refresh`
- **接口说明：** 使用刷新令牌换取新的访问令牌
- **是否鉴权：** 否 / 视实现而定

#### 请求体

```json
{
  "refreshToken": "refresh-token"
}
```

#### 业务约束

- 第一版可先预留，不强制实现；
- 若实现，则需结合 `auth_refresh_token` 表管理令牌有效性。

### 10.2.2 修改个人信息

- **接口路径：** `PUT /api/user/profile`
- **接口说明：** 修改当前登录用户个人资料
- **是否鉴权：** 是

#### 请求体

```json
{
  "nickname": "樱同学",
  "realName": "Sakura",
  "gender": 2,
  "email": "new@example.com",
  "phone": "13800000000",
  "avatarUrl": "https://example.com/avatar.png",

```

#### 请求字段约束

| 字段名 | 类型 | 是否必填 | 约束说明 |
|---|---|---|---|
| `nickname` | String | 是 | 1~50 位 |
| `realName` | String | 否 | 0~50 位 |
| `gender` | Integer | 否 | 0、1、2 |
| `email` | String | 否 | 合法邮箱格式，若填写则唯一 |
| `phone` | String | 否 | 合法手机号格式，若填写则唯一 |
| `avatarUrl` | String | 否 | 长度不超过 255 |

#### 业务约束

- 只能修改当前用户资料；
- 邮箱、手机号若填写则需做唯一校验；
- 不允许通过此接口修改用户名、账号状态、逻辑删除标记。

---

### 10.2.3 修改密码

- **接口路径：** `PUT /api/user/password`
- **接口说明：** 修改当前登录用户登录密码
- **是否鉴权：** 是

#### 请求体

```json
{
  "oldPassword": "12345678",
  "newPassword": "abcd1234",
  "confirmPassword": "abcd1234"
}
```

#### 业务约束

- 原密码必须正确；
- 新密码与确认密码必须一致；
- 新密码不能与旧密码完全相同；
- 修改成功后建议使旧令牌失效或提示重新登录。

## 10.3 学习计划模块接口

### 10.3.1 新建学习计划

- **接口路径：** `POST /api/study-plan`
- **接口说明：** 新增一条学习计划
- **是否鉴权：** 是

#### 请求体

```json
{
  "planTitle": "学习 Java 集合",
  "planContent": "重点看 List、Set、Map 的实现差异",
  "planDate": "2026-04-03",
  "startTime": "19:00:00",
  "endTime": "21:00:00",
  "priorityLevel": 1,
  "remark": "今晚必须完成"
}
```

#### 请求字段约束

| 字段名 | 类型 | 是否必填 | 约束说明 |
|---|---|---|---|
| `planTitle` | String | 是 | 1~100 位 |
| `planContent` | String | 否 | 不超过合理长度 |
| `planDate` | String | 是 | 格式为 yyyy-MM-dd |
| `startTime` | String | 否 | 格式为 HH:mm:ss |

## 13. 开发实现建议

### 13.1 Controller 层建议

- 仅负责接收参数、参数校验、返回结果；
- 不直接写复杂业务逻辑；
- 统一通过 `ApiResponse` 返回。

### 13.2 Service 层建议

- 处理业务逻辑；
- 校验当前数据是否归属当前用户；
- 处理唯一性、状态流转、逻辑删除等规则。

### 13.3 Mapper 层建议

- 统一使用 MyBatis；
- SQL 建议写在 XML 中；
- 列表查询需默认带 `deleted_flag = 0` 条件；
- 所有业务查询应带 `user_id = 当前用户ID` 条件。
